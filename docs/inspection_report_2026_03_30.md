# AIOS Kernel 프로젝트 종합 점검 보고서

**작성일:** 2026년 3월 30일  
**대상 저장소:** `tjwlstj/aios-kernel`  
**작성자:** Manus AI  

## 1. 프로젝트 개요 및 현황 요약

`aios-kernel`은 AI 워크로드에 최적화된 x86_64 기반의 베어메탈 운영체제 커널 프로토타입입니다. 이 프로젝트는 텐서 중심의 메모리 관리, AI 워크로드 전용 스케줄러, 가속기 하드웨어 추상화 계층(HAL), 그리고 시스템 자율 제어(Autonomy) 평면을 포함하는 도전적이고 혁신적인 아키텍처를 시도하고 있습니다. 

최근 커밋 이력과 코드를 분석한 결과, 2026년 3월 22일자 "점검 및 부족한 점 리포트"에서 지적되었던 여러 한계점들이 적극적으로 개선되고 있음을 확인했습니다. 특히 안정성을 위한 헬스 게이트(Health Gate) 도입, 부팅 시 메모리 셀프 테스트, 공통 시간원 통합, 그리고 GitHub Actions를 통한 CI/CD 파이프라인 구축 등 인프라 측면에서 상당한 진전이 있었습니다.

그러나 여전히 코드 수준에서는 커널 패닉을 유발할 수 있는 잠재적 버그(NULL 역참조, 오버플로우)와 멀티코어 환경을 대비한 동시성 제어 메커니즘의 부재 등 해결해야 할 과제들이 남아 있습니다.

## 2. 이전 점검 대비 주요 개선 사항

2026년 3월 22일 리포트에서 지적된 우선 조치 사항들이 다음과 같이 반영되었습니다.

| 지적 사항 | 개선 현황 | 관련 파일/모듈 |
| :--- | :--- | :--- |
| **검증 체계 부재** | GitHub Actions 기반의 Linux Boot Check CI 워크플로우가 추가되었으며, QEMU 스모크 테스트(`make test`)가 도입되어 부팅 로그 시그니처 검증이 자동화되었습니다. | `.github/workflows/linux-boot-check.yml`, `Makefile` |
| **스케줄러 시간원 단순화** | PIT/TSC 기반의 단조 증가 나노초 시간원(`kernel_time_init`)이 통합되어 스케줄러와 자율 제어 평면이 공통된 시간 기준을 사용하게 되었습니다. | `kernel/time.c`, `include/kernel/time.h` |
| **HAL 실 디바이스 경로 부족** | ACPI 및 PCI Core 스캐닝이 강화되었고, e1000 네트워크 부트스트랩, USB 및 스토리지 호스트의 최소 탐색 골격이 추가되었습니다. | `drivers/pci_core.c`, `drivers/e1000.c` 등 |
| **자율제어 평가 미완성** | `compute_eval_score` 및 롤백 트리거 로직이 추가되어, 텔레메트리 프레임 간의 변화를 기반으로 자율 제어의 효과를 평가하고 롤백하는 구조가 구현되었습니다. | `runtime/autonomy.c` |
| **안정성 게이트 부재** | 서브시스템별 상태를 추적하는 Health Registry(`kernel_health.c`)와 부팅 시 메모리 성능을 측정하는 Selftest 모듈이 추가되어 시스템 안정성 기반의 자율성 제어가 가능해졌습니다. | `kernel/health.c`, `kernel/selftest.c` |

## 3. 핵심 서브시스템 코드 리뷰 및 취약점 분석

15개의 핵심 C/ASM 소스 파일(총 약 6,500줄 분량)에 대한 정적 분석 및 병렬 리뷰를 수행한 결과, 다음과 같은 주요 이슈들이 도출되었습니다.

### 3.1. 잠재적 크래시 및 메모리 안전성 (Critical Issues)

가장 시급하게 해결해야 할 문제는 **NULL 포인터 역참조**와 **정수 오버플로우** 가능성입니다. 베어메탈 커널 환경에서 이러한 버그는 즉각적인 시스템 패닉으로 이어집니다.

*   **시스템 콜 입력 검증 부재:** `runtime/ai_syscall.c`에서 유저 공간으로부터 전달받은 포인터(예: `req`, `stats`)에 대한 주소 공간 유효성 검증(`access_ok` 등)이 누락되어 있습니다. 또한, 모델 레지스트리가 가득 찼을 때 경계를 벗어난 쓰기가 발생할 수 있는 버퍼 오버플로우 취약점이 존재합니다.
*   **할당 실패 및 반환값 검증 누락:** `hal/accel_hal.c`의 `accel_mem_alloc` 등 여러 함수에서 포인터 인자나 내부 함수 반환값에 대한 NULL 체크 없이 역참조를 시도하는 구간이 다수 발견되었습니다.
*   **산술 오버플로우:** `mm/tensor_mm.c`의 `tensor_total_size` 계산, `sched/ai_sched.c`의 런타임 계산, `kernel/selftest.c`의 대역폭 연산 시 64비트 정수 한계를 초과할 위험이 존재합니다. 안전한 산술 연산 매크로나 라이브러리 도입이 필요합니다.

### 3.2. 동시성 제어 및 락(Lock) 부재

현재 커널 코드는 인터럽트 활성화/비활성화(`cli`, `sti`) 외에는 스핀락(Spinlock)이나 뮤텍스(Mutex)와 같은 명시적인 동시성 보호 메커니즘이 전무합니다. 

*   `kernel/health.c`의 전역 배열 `g_health` 업데이트, `mm/tensor_mm.c`의 메모리 풀 접근, 스케줄러 큐 조작 등에서 경쟁 상태(Race Condition)가 발생할 수 있습니다. 
*   단일 코어 환경을 가정한 프로토타입 단계라면 문제가 없으나, AI 워크로드를 위한 다중 코어 확장성을 고려할 때 가장 먼저 아키텍처에 반영해야 할 부분입니다.

### 3.3. 코드 품질 및 유지보수성 (Warnings)

*   **매직 넘버 남용:** PCI 클래스 코드(`0xFC`, `0x80`), 가속기 제어 레지스터 주소, 텐서 할당 리스트 크기(8), VGA 레지스터 포트(`0x3D4`) 등 의미를 알기 어려운 매직 넘버가 코드 전반에 산재해 있습니다. 이들은 모두 `#define` 매크로나 `enum`으로 상수화해야 합니다.
*   **하드코딩된 로직:** `platform_probe.c`의 우선순위 분류나 `accel_hal.c`의 NVIDIA 기능(Capability) 설정 등이 하드코딩되어 있어, 새로운 하드웨어나 모델 아키텍처를 추가할 때 확장성이 떨어집니다.

### 3.4. 우수한 설계 요소 (Positive Aspects)

*   **명확한 모듈화 및 네이밍:** `snake_case` 네이밍 컨벤션이 일관되게 적용되어 있으며, 각 파일이 담당하는 도메인(예: `pci_core`, `tensor_mm`)이 명확하게 분리되어 가독성이 뛰어납니다.
*   **자율 제어 상태 머신:** `autonomy.c`에서 제안(Propose) -> 승인(Approve) -> 실행(Commit) -> 롤백(Rollback)으로 이어지는 상태 전이 파이프라인은 AI 운영체제라는 컨셉에 매우 부합하는 훌륭한 설계입니다.
*   **Fallback 메커니즘:** 전용 가속기가 없을 경우 CPU SIMD 명령어를 활용하는 Fallback 구조(`cpu_simd_ops`)를 HAL 계층에 마련한 점은 시스템 범용성을 크게 높입니다.

## 4. 향후 개선 권고 사항 (Recommendations)

프로젝트를 단순한 아키텍처 데모에서 상용화 가능한 안정적인 커널 수준으로 끌어올리기 위해 다음 단계들을 권고합니다.

1.  **동시성 제어 원시 타입(Primitives) 구현:**
    *   `spinlock_t` 및 `mutex_t` 자료구조와 획득/해제 어셈블리 루틴(`xchg`, `cmpxchg` 활용)을 구현하고, 전역 상태를 변경하는 모든 서브시스템(메모리, 스케줄러, 헬스 레지스트리)에 적용해야 합니다.
2.  **안전한 시스템 콜 경계 구축:**
    *   사용자 공간과 커널 공간의 메모리를 엄격히 분리하고, 시스템 콜 핸들러에서 사용자 포인터를 검증하는 `copy_from_user` / `copy_to_user` 헬퍼 함수를 도입해야 합니다.
3.  **정적 분석 및 린터 CI 통합:**
    *   현재의 빌드/스모크 테스트 CI에 더해, `cppcheck`나 `clang-tidy`와 같은 정적 분석 도구를 GitHub Actions에 연동하여 NULL 역참조와 매직 넘버 사용을 커밋 단계에서 차단할 것을 권장합니다.
4.  **에러 전파 아키텍처 강화:**
    *   에러 발생 시 단순히 `return;`으로 종료하거나 로그만 남기는 대신, 모든 내부 함수가 `aios_status_t`를 반환하도록 리팩토링하여 상위 호출자(특히 Autonomy Control Plane)가 실패 원인을 정확히 인지하고 대응할 수 있도록 해야 합니다.

## 5. 결론

`tjwlstj/aios-kernel`은 AI 중심의 자율 운영체제라는 비전을 구체적인 코드로 실체화하고 있는 매우 흥미로운 프로젝트입니다. 지난 점검 이후 인프라와 아키텍처 측면에서 큰 발전이 있었으며, 핵심 뼈대는 성공적으로 완성되었습니다. 이제 다음 마일스톤은 "안정성과 견고함"입니다. 동시성 제어, 포인터 검증, 매직 넘버 제거 등의 코드 품질 개선을 통해 커널의 신뢰성을 확보한다면, 로드맵에 명시된 분산 처리 및 실 하드웨어 연동 목표를 성공적으로 달성할 수 있을 것입니다.
